# 播放、推荐与广告追踪设计

## 1. 按来源选集与播放会话

```text
用户选择来源 → 获取该来源线路/剧集 → 选择 SourceEpisode
  → FastAPI 校验用户、全局来源和用户选择
  → 解析/确认 MediaVersion → Redis 创建短期播放会话
  → 返回内部 manifest 地址 → hls.js 请求内部资源
```

来源切换只刷新该来源的实际剧集。若 SourceEpisode 映射到同一标准剧集，前端可携带恢复意图，服务端返回标准进度；无可靠映射时明确要求确认。

## 2. 媒体版本

广告时间不能只绑定影视标题或源地址。`MediaVersion` 用稳定但脱敏的指纹区分内容版本，候选输入包括：

- 规范化清单结构与分片时长序列摘要。
- 总时长和主清晰度集合。
- 来源条目、线路和剧集内部 ID。
- 可用的上游稳定版本字段。

重定向 URL、签名参数和临时 token 不进入指纹原文。指纹策略必须版本化，以便重新计算和解释失效。

## 3. 广告标记流程

```text
注册用户点击开始/结束
  → 客户端生成草稿并允许微调
  → API 验证账户、声明、访问权、媒体版本和区段范围
  → PostgreSQL 保存 personal_active 标记
  → ARQ 聚合相同 MediaVersion 的近似区段
  → 形成 candidate / verified / disputed
  → 播放器只对 verified 区段提供可信自动跳过
```

个人标记可立即供本人手动跳过。聚合必须考虑区段重合度、提交账户质量、错误报告和版本一致性，不能简单按提交人数投票。具体阈值通过离线 fixture 和评测确定。

## 4. 跳过与撤销

- 播放器提前获取当前 MediaVersion 的有效区段。
- 进入区段时展示手动跳过；用户开启偏好且区段可信时自动 seek 到 `end_ms`。
- 跳过后短时保存 `previous_time`，允许撤销回到 `start_ms`。
- 撤销后当前区段本次播放不再自动跳过，避免循环。
- seek 失败、直播流、时长未知或区段越界时不执行并报告脱敏结果。

## 5. 同类推荐

候选生成：同内容类型 → 分类/标签交集 → 地区/年份相近 → 可播放来源覆盖。排序排除当前影视、不可播放、用户隐藏和已明确不感兴趣条目。

响应提供结构化理由：`same_genres`、`same_theme`、`regional_popular`、`provider_trending`。前端把理由翻译为用户文案。推荐服务不可直接创建播放会话。

## 6. 隐私与滥用控制

- 标记 API 按用户、媒体版本和时间窗口限流。
- 管理员看聚合和争议摘要，不看其他用户完整观看轨迹。
- 异常大量、重叠、越界或机械化标记降低可信度并进入安全事件摘要。
- 日志和审计只记录内部 ID、区段与结论，不记录完整媒体 URL。

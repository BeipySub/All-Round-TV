# 项目目录结构与职责

## 设计原则

- 应用、共享包、后端服务、基础设施、文档和历史代码明确隔离。
- 目录名称表达业务职责，不以临时任务或开发者姓名命名。
- 旧版归档不参与新版构建。
- 源码目录只放源码；方案、记录和原型统一进入 `docs`。

## 一级目录

| 目录 | 作用 | 禁止内容 |
| --- | --- | --- |
| `apps` | 可独立构建的用户应用 | 后端业务、部署秘密 |
| `packages` | 前端共享工作区包 | 页面业务、数据库代码 |
| `services` | 后端 API 与 Worker | 前端页面、原型文件 |
| `infra` | Docker、Nginx、数据库辅助和监控 | 业务逻辑、真实密钥 |
| `docs` | 架构、方案、记录、AI 经验、原型 | 构建产物、敏感数据 |
| `scripts` | 可重复执行的维护脚本 | 一次性且无说明的临时代码 |
| `tests` | 跨模块测试 | 单模块私有测试 |
| `legacy` | 旧版只读归档 | V2 新功能 |

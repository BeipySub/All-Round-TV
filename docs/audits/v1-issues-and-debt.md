## 2. 性能与可靠性问题

### 2.1 搜索请求风暴

每个已选源最多请求 50 页，所有页和所有源并行。换源又复用完整搜索。没有全局并发限制、分页取消、后端缓存或用户请求预算。

### 2.2 频繁 localStorage 写入

播放进度可被 5 秒 timeupdate、30 秒 interval、pause、visibilitychange 和 beforeunload 重复触发。每次可能序列化完整 50 条历史和完整剧集数组。

### 2.3 大响应缓冲

Serverless 代理把所有内容读入字符串，并可能 JSON 包装后写入 KV；大媒体或异常响应会造成内存、执行时间和缓存额度问题。

### 2.4 无稳定取消传播

前端 AbortController 只能中止浏览器到代理的请求，服务端是否取消上游请求没有统一保证。Promise.race 的超时也没有总是中止原 fetch。

## 3. 架构技术债

- 五个主要 JavaScript 文件合计超过 5500 行，职责高度混合。
- 依赖通过脚本顺序、全局变量和 DOM ID 隐式建立。
- 覆盖 `window.fetch` 模拟 API，接口边界不可观察。
- 首页和播放器重复加载并复用不相关模块。
- 同一功能存在多套实现：剧集控制、历史保存、聚合搜索、代理、HTML 注入。
- 多个配置对象仅声明未执行，形成“伪安全/伪抽象”。
- 大量 catch 块静默吞掉错误，无法区分预期降级与程序缺陷。
- UI、业务、网络、存储和安全代码混在同一文件。
- 没有数据库、领域模型、迁移或审计。

## 4. 测试与质量债

- 没有单元、集成、端到端、安全或媒体 fixture 测试。
- 没有 TypeScript 或第一方运行时 schema。
- 没有 Lint、格式化或提交前检查。
- 第一方脚本虽通过语法检查，但没有验证脚本顺序与 DOM 契约。
- 四套部署实现没有契约测试，实际功能差异长期积累。
- 没有 HLS 主清单、KEY、MAP、Range、重定向和二进制测试。

## 5. 可维护性问题

- 内联事件和动态 HTML 使重命名、拆分和自动补全困难。
- 中英文注释和历史修复说明混杂，存在大量被注释掉的旧路径。
- `config.js` 同时管理站点、资源源、播放器、搜索、安全和自定义源。
- 资源源配置直接提交到浏览器，无法集中审核、禁用和监控。
- 第三方压缩库无法从 package lock 判断实际版本与补丁。
- 版本检查依赖已归档上游仓库和第三方 GitHub 代理。

## 6. 产品逻辑债

- 豆瓣条目只是搜索入口，推荐内容与 CMS 资源没有统一影视实体。
- 搜索结果按标题排序而非相关度，没有跨源合并。
- 详情只使用 CMS 第一个播放组。
- 换源只按标题匹配，未考虑年份、类型、季和版本。
- 历史保存资源直链而非稳定条目/剧集引用。
- 成人内容过滤是前端关键词列表，不能构成可靠治理。
- 资源源由所有用户任意添加，没有审核、健康检查或责任边界。

## 7. 不应迁移的旧实现

- 共享密码哈希鉴权。
- 任意 URL 代理。
- 全局 fetch 覆盖。
- watch.html 固定延迟跳转。
- localStorage 传递完整剧集和媒体 URL。
- 删除 discontinuity 的“广告过滤”。
- 复制全部原始播放直链。
- 多平台分叉代理。
- 自动提交时间戳版本。
- 外部 AllOrigins 备用代理。

## 8. 可保留的用户价值

- 多源局部失败不阻断整体搜索。
- 推荐内容快速触发全源搜索。
- 历史恢复时刷新最新剧集，失败使用缓存。
- 播放器上一集/下一集、自动连播、排序、快捷键和移动端交互。
- 换源时尽量保持当前集数。
- 设置与历史的导入导出价值。

这些价值应保留，但必须用 V2 的标准影视模型、权限、后端聚合和安全播放会话重新实现。

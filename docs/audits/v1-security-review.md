# V1 安全审查

> 本报告基于静态代码。严重度表示在私有多用户 V2 场景中的优先级，不表示已经发生实际攻击。

## 1. 风险摘要

| 等级 | 数量 | 结论 |
| --- | ---: | --- |
| P0 / 严重 | 3 | V2 开发前必须从架构层消除，禁止兼容保留 |
| P1 / 高 | 5 | 身份、输入和媒体设计阶段必须解决 |
| P2 / 中 | 7 | 工程化与上线前解决 |

## 2. P0：浏览器获得等价代理凭据

服务器将共享密码的 SHA-256 哈希注入 HTML。代理直接比较该哈希，因此哈希本身就是可重放的代理凭据，不只是“不可逆的密码摘要”。

进一步问题：

- 哈希保存在 `passwordVerified` 和 `proxyAuthHash`。
- 哈希通过 `auth` 查询参数发送，可能进入浏览器历史、访问日志、反向代理和监控。
- 时间戳不是必填；只提供正确哈希即可永久使用。
- 时间戳只拒绝太旧值，不拒绝未来值。
- 凭据没有绑定请求方法、目标 URL 或用户。
- Express 鉴权失败日志会输出预期哈希与收到哈希。

证据：

- `js/password.js:46-80`
- `js/proxy-auth.js:12-103`
- `server.mjs:121-150`
- `api/proxy/[...path].mjs:306-335`

V2 处理：完全删除此协议。使用服务端随机会话 Cookie；代理只接受由已登录用户基于已审核资源创建的短期播放会话。

## 3. P0：任意 URL 代理与 SSRF

拥有共享哈希的浏览器可以构造 `/proxy/<任意 URL>`。Serverless 实现只检查 HTTP/HTTPS 格式；Express 仅对字面 hostname 与少数字符串前缀做判断。

可绕过面包括：

- DNS 解析到私网或回环地址。
- 外部地址重定向到私网。
- IPv6、IPv4-mapped IPv6 和非标准 IP 表示。
- 云平台元数据地址。
- DNS 重绑定。
- 当前黑名单未覆盖的保留、链路本地和特殊用途网段。

所有实现都跟随重定向，且重定向后不重新验证地址。

影响：读取部署环境可访问的内部 HTTP 服务、管理接口或云元数据；在多用户系统中不可接受。

证据：`server.mjs:97-118`、三套 Serverless 的 `getTargetUrlFromPath` 与 `fetchContentWithType`。

V2 处理：不存在通用代理接口；所有出站目标来自已审核资源与 Provider。统一客户端在每次解析和每次重定向后验证全部 IP，并执行协议、端口、响应大小、超时和并发限制。

## 4. P0：动态 HTML 注入与持久化 XSS

上游数据、用户自定义源和导入历史被拼接进 `innerHTML` 与内联事件。现有转义不完整且不同字段规则不一致。

示例：

- `vod_name` 未转义单引号，却放入单引号包裹的 `onclick` 参数。
- `source_name`、详情字段、图片 URL 与部分类型字段直接进入 HTML。
- 观看历史的 `item.url` 直接进入 `onclick`。
- 远程或本地配置导入可写入历史和任意 localStorage key。
- 自定义 API 名称、URL 和 detail 会参与动态页面生成。

证据：

- `js/app.js:731-800, 903-970`
- `js/ui.js:393-461`
- `js/app.js:1226-1281`

V2 处理：Vue 模板默认文本绑定；禁止将不可信内容传给 `v-html`；URL 使用后端标准化 ID；导入执行严格 schema 和字段白名单；增加 XSS 测试 fixture。

## 5. P1：配置文件校验不提供真实性

配置文件中的 SHA-256 只证明 data 与 hash 相符，攻击者可以任意修改并重新计算。导入代码把 `data` 的所有 key 写入 localStorage，没有白名单。

本地文件最大 10 MB，远程配置没有明确响应大小限制。`cfgVer` 不参与版本逻辑。

证据：`js/app.js:1141-1336`。

V2 处理：上传后端；限制总大小；解析固定版本；逐字段校验；预览后确认；未知字段忽略；敏感字段禁止导入；记录导入批次。

## 6. P1：认证与账户控制不足

- 所有人共享同一个密码，无用户身份和角色。
- 验证可持续 90 天，没有单会话撤销。
- 没有登录失败限流或审计。
- 修改密码只能通过部署环境变量，无法管理现有浏览器状态之外的访问。
- SHA-256 是快速哈希，不适合密码存储。

V2 处理：Argon2id、管理员邀请制、用户状态、Redis 会话、设备会话撤销、限流和审计。

## 7. P1：媒体地址与用户网络隐私

主播放流程把媒体直链放在浏览器 URL、历史和 localStorage，并由浏览器直接连接第三方媒体服务器。

影响：

- 上游获得用户 IP、User-Agent 和可能的 Referer。
- 带签名的播放地址可能泄漏到历史、截图、日志或分享链接。
- 地址过期后历史无法稳定恢复。
- 浏览器直接暴露于第三方内容与证书/CORS 行为。

V2 处理：短期播放会话和内部资源 token；浏览器不接触原始 URL；日志只记录内部 ID。

## 8. P1：缺少请求限流与资源约束

- 搜索可对每个源同时请求最多 50 页。
- 多源搜索和换源均可触发请求风暴。
- Serverless 代理读取整个响应，不限制正文大小。
- 没有按用户、IP、源或目标域名的并发与速率限制。
- 自定义源最多 5 个只在部分逻辑中生效，不能保护代理接口。

V2 处理：Redis 限流、每用户并发预算、Provider/CMS 独立熔断、分页上限、响应大小限制和后台任务队列。

## 9. P1：Serverless 二进制与认证链路缺陷

- 非 M3U8 内容统一 `response.text()`，二进制会损坏。
- M3U8 重写的分片、KEY、MAP 地址没有认证参数，后续请求会被鉴权拒绝。
- 返回错误时可能暴露目标 URL、上游错误正文和堆栈。

这既是稳定性问题，也是信息泄漏和资源消耗风险。

## 10. P2 风险

### 10.1 CORS 过宽

所有代理默认允许 `Access-Control-Allow-Origin: *`，且多个实现允许全部请求头。共享哈希一旦泄漏，可从任意站点调用。

### 10.2 安全响应头不足

Express 仅设置 `nosniff`、`SAMEORIGIN` 和已过时的 `X-XSS-Protection`。缺少明确 CSP、Referrer-Policy、Permissions-Policy 和 HSTS 部署策略。Serverless 路径也不统一。

### 10.3 日志泄漏

Vercel 日志输出完整 URL 和 query；错误响应包含目标 URL；Express 会打印哈希。Provider 与媒体 URL 可能包含敏感查询参数。

### 10.4 外部备用代理

豆瓣失败时将目标 URL交给 `api.allorigins.win`。该第三方不受项目控制，会增加隐私、可用性和供应链风险。

### 10.5 供应链不可审计

ArtPlayer、hls.js、Tailwind 和 js-sha256 以压缩文件提交，缺少锁定来源、完整性元数据、自动漏洞扫描和升级记录。

### 10.6 直接展示第三方图片

搜索和换源卡片直接加载上游图片，暴露用户网络信息，且缺少统一图片代理、域名策略和内容限制。

### 10.7 安全开关仅为声明

`SECURITY_CONFIG.enableXSSProtection`、`sanitizeUrls`、`maxQueryLength` 没有形成统一执行层，容易让维护者误以为保护已经启用。

## 11. V2 安全验收门槛

- 不存在浏览器可调用的任意 URL 代理。
- 用户、邀请、会话和 CSRF 均有自动化攻击测试。
- 出站客户端覆盖 DNS 重绑定、全部私网/保留地址、重定向和响应限制。
- 媒体清单和分片只使用短期内部 token。
- 日志脱敏测试确认不出现密码、Cookie、邀请 token、上游签名 URL。
- 导入数据使用严格 schema 并经过预览确认。
- 所有动态内容使用安全渲染，XSS fixture 通过。
- 依赖来自锁文件和包管理器，并进入漏洞扫描。
